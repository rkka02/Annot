# Annot : PDF Research Assistant — 프로젝트 계획서


## 1. 프로젝트 개요

### 목적
논문을 읽으며 동시에 AI와 대화할 수 있는 웹 기반 도구를 만든다.
VSCode가 "코드 편집기 + AI 채팅"의 조합이라면, 이 프로젝트는 **"PDF 리더 + AI 채팅"의 조합**이다.

### 해결하려는 문제
- 현재 Claude/ChatGPT에 PDF를 업로드하면, PDF 원문과 채팅창을 동시에 볼 수 없다.
  원문을 확인하려면 앱을 왔다갔다 해야 하고, 맥락이 끊긴다.
- 매번 새 대화를 시작하면 이전에 했던 질문과 맥락이 사라진다.
- 논문을 읽다가 떠오르는 질문을 즉시, 흐름이 끊기지 않는 환경에서 던지고 싶다.

### 핵심 가치
PDF를 **보면서** AI에게 질문하고, 그 대화가 **세션으로 누적**되어 다음에 이어서 쓸 수 있는 환경.

## 2. 설계 논의 과정

### 2.1 초기 아이디어
- PDF 뷰어 + AI 채팅 패널이 나란히 배치된 인터페이스.
- 논문 읽으면서 중요한 부분은 노란색 하이라이트, 모르는 부분은 빨간색 하이라이트로 개인 정리.
- 모르는 부분이 있으면 옆 채팅창에 질문.

### 2.2 컨텍스트 범위에 대한 논의
- **프로젝트/워크스페이스 단위**로 이전 논문, 메모, 대화 이력을 참조하여 답변하는 구상이 있었다.
- Logseq 등 메모 앱과의 연동도 논의되었으나, 현 시점에서는 시기상조로 판단.
- **MVP에서는 제외**하고, 현재 PDF + 대화 이력만으로 시작하기로 결정.

### 2.3 RAG 관련 논의
- 프로젝트에 자료가 쌓이면 컨텍스트 윈도우에 다 넣을 수 없으므로 RAG이 필요.
- 단, RAG은 파편화된 정보만 가져오고 유기적 맥락은 놓치는 한계가 있음.
- **해결 아이디어**: RAG top-K 결과 + Logseq 그래프의 연결된 노드를 함께 가져오는 2단계 검색.
- 이 역시 MVP 이후 단계로 보류. 현재는 세션 기반 대화 이력으로 컨텍스트를 직접 관리.

### 2.4 AI 호출 방식
- CLI 호출 (codex/claude code non-interactive) vs OAuth key API 직접 호출을 비교.
- **OAuth key 방식 채택**: 프로세스 오버헤드 없이 HTTP 요청으로 가볍게 호출 가능. 대화 이력을 messages 배열로 직접 관리하여 세션 구현이 깔끔. 구독 내 사용으로 추가 비용 없음.

### 2.5 플랫폼 결정
- Mac, Windows, iPad 등 기기를 자유롭게 오가며 사용할 수 있어야 함.
- **웹앱**으로 결정. 브라우저만 있으면 어디서든 접속 가능.
- 맥미니를 셀프호스팅 서버로 사용.

## 3. MVP 스펙

### 기능
| 기능 | 설명 |
|------|------|
| PDF 뷰어 | pdf.js 기반. 논문을 브라우저에서 렌더링 |
| 하이라이트 | 노란색(중요), 빨간색(모름) 하이라이트. 개인 정리용, AI와 공유하지 않음 |
| AI 채팅 패널 | PDF 옆에 채팅창. 질문 입력 → Claude, GPT 응답 |
| 세션 관리 | 대화 이력을 서버에 저장. 기기를 바꿔도 이어서 사용 가능 |
| 컨텍스트 참조 | 현재 열린 PDF 내용 + 현재 세션의 대화 이력만 참조 |

### 기술 스택
| 구성 요소 | 기술 |
|-----------|------|
| 프론트엔드 | 웹앱 (PDF 뷰어 + 채팅 UI) |
| PDF 렌더링 | pdf.js |
| AI API | Claude, GPT API (OAuth key) |
| 서버 | 맥미니 셀프호스팅 |
| 데이터 저장 | 서버 로컬 (PDF 파일, 세션 이력) |
