# Annot : PDF Research Assistant — 프로젝트 계획서


## 1. 프로젝트 개요

### 목적
논문을 읽으며 동시에 AI와 대화할 수 있는 도구를 만든다.
VSCode가 "코드 편집기 + AI 채팅"의 조합이라면, 이 프로젝트는 **"PDF 리더 + AI 채팅"의 조합**이다.

### 해결하려는 문제
- 현재 ChatGPT에 PDF를 업로드하면, PDF 원문과 채팅창을 동시에 볼 수 없다.
  원문을 확인하려면 앱을 왔다갔다 해야 하고, 맥락이 끊긴다.
- 매번 새 대화를 시작하면 이전에 했던 질문과 맥락이 사라진다.
- 논문을 읽다가 떠오르는 질문을 즉시, 흐름이 끊기지 않는 환경에서 던지고 싶다.

### 핵심 가치
PDF를 **보면서** AI에게 질문하고, 그 대화가 **세션으로 누적**되어 다음에 이어서 쓸 수 있는 환경.

## 2. 설계 논의 과정

### 2.1 초기 아이디어
- PDF 뷰어 + AI 채팅 패널이 나란히 배치된 인터페이스.
- 논문 읽으면서 중요한 부분은 노란색 하이라이트, 모르는 부분은 빨간색 하이라이트로 개인 정리.
- 모르는 부분이 있으면 옆 채팅창에 질문.

### 2.2 컨텍스트 범위: 트리 기반 스코핑
- 초기에는 "프로젝트 단위"로 이전 논문, 메모, 대화 이력을 참조하는 구상이 있었다.
- 이후 **트리(폴더) 구조**로 발전. 리서치 자료를 계층적으로 구조화하고, 세션도 노드 성격에 따라 나뉜다.
- **핵심 규칙**: 세션은 두 종류다.
  - **폴더 세션**: 해당 폴더와 하위 PDF들을 아우르는 포괄적 리서치 세션.
  - **PDF 세션**: 특정 PDF 리프 노드에 귀속되는 개별 읽기 세션.
  - 즉 폴더에서는 넓은 논의를 하고, PDF를 읽으면서 묻는 질문은 기본적으로 그 PDF 전용 세션으로 분리한다.
  - 현재 세션 폴더와 그 하위 PDF는 **기본 컨텍스트/우선 탐색 대상**이 된다.
  - 다만 스코프는 엄격한 보안 경계가 아니라 **느슨한 작업 힌트**로 취급한다.
  - Codex는 필요하다고 판단하면 다른 폴더의 PDF도 탐색할 수 있다.
  - 즉 트리는 자료를 정리하고 세션을 구분하는 구조이며, 모델 접근 자체를 강하게 제한하지는 않는다.
  - PDF는 해당 세션 모델이 접근 가능한 **리소스**로 기능한다.

#### 예시
```
My Research/
├── 📁 Machine Learning          ← 폴더 세션: ML 자료를 넓게 논의
│   ├── 📄 survey-2024.pdf
│   ├── 📁 Transformers
│   │   ├── 📄 attention.pdf    ← PDF 세션: attention.pdf 읽기 대화 전용
│   │   └── 📄 bert.pdf         ← PDF 세션: bert.pdf 읽기 대화 전용
│   └── 📁 Reinforcement Learning
│       └── 📄 ppo-paper.pdf
└── 📁 Physics                   ← 필요시 이쪽도 탐색 가능
    └── 📄 quantum.pdf
```

- 사용성 기준:
  - 폴더에서 "Open Research Session"을 누르면 **폴더 세션**이 열린다.
  - PDF를 열고 채팅을 시작하면, 기본적으로 그 PDF에 귀속되는 **PDF 세션**이 열린다.
  - 사용자는 "폴더 단위로 묶인 토론"과 "논문 한 편을 읽으며 하는 질의응답"을 분리해서 보존할 수 있어야 한다.

### 2.3 PDF 컨텍스트 전달: Tool Use 방식
- 기존 접근: PDF 텍스트를 미리 추출해서 시스템 프롬프트에 삽입 → 컨텍스트 낭비, 스케일 안 됨.
- **Codex 프로세스 방식 채택 이후 재정리**:
  - 별도 `read_pdf(path, pages?)` tool을 Annot이 직접 구현하지 않는다.
  - 대신 세션별 Codex 프로세스가 로컬 파일시스템에 직접 접근해 PDF를 읽는다.
  - Annot은 "현재 세션 폴더", "현재 사용자가 보고 있는 PDF", "우선적으로 참고할 자료" 정도만 힌트로 제공한다.
  - Codex는 필요하면 직접 파일을 열고, 다른 도구를 시도하고, 텍스트가 깨진 PDF에도 자율적으로 대응한다.
- **장점**: Annot 쪽 tool 구현 부담이 줄고, Codex가 실제 로컬 연구 폴더에 붙어 있는 것처럼 동작한다.
- 세션 종류에 따른 기본 힌트:
  - 폴더 세션: 현재 폴더와 하위 PDF들을 우선적으로 참조
  - PDF 세션: 현재 PDF를 최우선 컨텍스트로 두고, 필요하면 같은 폴더/워크스페이스 자료를 추가 참조

### 2.4 RAG 관련 논의 (MVP 이후)
- 자료가 더 쌓이면 RAG이 필요할 수 있음.
- RAG은 파편화된 정보만 가져오고 유기적 맥락은 놓치는 한계가 있음.
- **해결 아이디어**: RAG top-K 결과 + 그래프의 연결된 노드를 함께 가져오는 2단계 검색.
- 현재는 Codex 프로세스 + 로컬 파일 접근으로 충분하므로 보류.

### 2.5 AI 호출 방식: Codex non-interactive exec + resume
- Annot은 Codex CLI의 **non-interactive mode**(`codex exec`)를 백엔드 실행 엔진으로 사용한다.
- 첫 요청에서는 새 Codex 세션을 시작하고, 이후 요청은 `codex exec resume <session_id>` 형태로 이어 붙인다.
- 즉 **상태는 Codex 세션에 유지**되고, Annot은 폴더 세션/PDF 세션 각각에 대응하는 Codex session id를 로컬 메타데이터에 저장해 재사용한다.
- 별도 API 키 불필요. ChatGPT 구독으로 사용. Codex 로그인 한 번이면 됨.
- Codex가 **파일시스템에 자율 접근** 가능 — PDF를 직접 읽고, 텍스트가 깨져도 스스로 대응 (다른 도구 시도, OCR 등).
- Annot이 별도 PDF tool layer를 구현할 필요 없이 Codex의 파일 접근/도구 사용 능력에 맡긴다.
- 사실상 **"연구 폴더에 Codex를 붙여놓는"** 것과 같은 경험을 유지하면서도, 상주 프로세스를 직접 관리할 필요는 줄어든다.
- 장점: HTTP처럼 매 요청마다 전체 메시지 이력을 재전송하지 않아도 되고, 장기 상주 프로세스의 idle timeout/lifecycle 관리 부담도 덜하다.
- 고려:
  - Annot은 폴더 세션과 PDF 세션 각각에 대해 Codex 세션 ID 매핑을 저장해야 한다.
  - `--ephemeral` 모드가 아니라 **persisted session** 기반으로 실행해야 resume이 가능하다.
  - Codex 세션 파일과 Annot 메타데이터가 어긋났을 때 복구 로직이 필요하다.

### 2.6 플랫폼 결정
- 현재는 Next.js 웹앱으로 개발 (빠른 프로토타이핑).
- **궁극적 목표**: 맥 네이티브 앱 (Tauri 또는 Electron). 맥북에서 로컬로 동작.
- 윈도우 빌드도 가능하지만, 우선순위는 macOS.

### 2.7 데이터 저장: 로컬 파일시스템
- 트리 구조를 로컬 폴더 구조로 그대로 미러링.
- PDF 파일은 실제 파일시스템 폴더에 배치. Finder에서도 직접 확인/관리 가능.
- 세션 이력, 하이라이트 등 메타데이터는 각 폴더에 `.annot/` 숨김 디렉토리로 저장한다.
- 세션 메타데이터는 최소 두 계층으로 나뉜다.
  - 폴더 세션: 해당 폴더의 `.annot/sessions.json`
  - PDF 세션: 해당 PDF를 키로 갖는 별도 세션 레코드 또는 `pdfSessions` 섹션
- 즉 PDF를 읽으며 한 질문은 같은 폴더 안의 다른 PDF 대화와 자동으로 섞이지 않아야 한다.

```
~/Annot/
├── Machine Learning/
│   ├── .annot/                  ← 메타데이터 (세션 이력, 하이라이트 등)
│   │   ├── sessions.json
│   │   └── highlights.json
│   ├── survey-2024.pdf
│   └── Transformers/
│       ├── .annot/
│       │   └── sessions.json
│       ├── attention.pdf
│       └── bert.pdf
└── Physics/
    ├── .annot/
    │   └── sessions.json
    └── quantum.pdf
```

## 3. MVP 스펙

### 기능
| 기능 | 설명 |
|------|------|
| 트리 탐색기 | 로컬 폴더 구조를 사이드바에 트리로 표시. 폴더/PDF 탐색 |
| PDF 뷰어 | pdf.js 기반. 논문을 브라우저에서 렌더링 |
| 하이라이트 | 노란색(중요), 빨간색(모름) 하이라이트. 개인 정리용 |
| AI 채팅 패널 | 폴더에서는 포괄적 폴더 세션을, PDF를 읽을 때는 해당 논문 전용 PDF 세션을 열어 대화 |
| 세션 관리 | 폴더 세션과 PDF 세션을 분리 저장. 다시 열면 각각의 Codex session id로 `codex exec resume` |
| 컨텍스트 참조 | 폴더 세션은 폴더 중심, PDF 세션은 현재 PDF 중심으로 힌트를 주고 Codex가 필요 자료를 직접 탐색 |
| 모델 선택 | Codex 백엔드에서 사용 가능한 모델 목록을 불러와 동적 선택 |

### 기술 스택
| 구성 요소 | 기술 |
|-----------|------|
| 프론트엔드 | Next.js 웹앱 (→ 추후 네이티브 앱) |
| PDF 렌더링 | pdf.js |
| AI API | OpenAI (Codex 백엔드, OAuth 토큰 재사용) |
| 데이터 저장 | 로컬 파일시스템 (폴더 = 트리, `.annot/` = 메타데이터) |

### 레이아웃
- **좌측 사이드바**: 트리 탐색기 (폴더/PDF 계층 구조)
- **중앙**: PDF 뷰어 (선택한 PDF 렌더링)
- **우측**: AI 채팅 패널
  - 폴더를 선택했을 때: 폴더 세션
  - PDF를 읽으며 열었을 때: 해당 PDF 세션
